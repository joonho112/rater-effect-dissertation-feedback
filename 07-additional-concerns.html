The issues gathered here are smaller in scope than those in the earlier chapters — think of it as housekeeping. Four independent problems, each with a clear fix, none of which threatens the dissertation’s central argument. Sorting them out before submission makes the manuscript sharper and forestalls the kind of reviewer queries that slow a review without adding anything substantive.

7.1 The condition count doesn’t add up (a reproducibility flag)

What happens. Table 18 (p. 84) defines the Study 3 grid along four factors: sample size (50, 100, 500 — three levels), data-collection design (complete, incomplete, anchor — three), effect type (none, severity, misfit — three), and rater correlation (0.2, 0.6, 0.9 — three). That is 3 × 3 × 3 × 3 = 81 conditions. Yet the text says “we created 108 combinations of simulation conditions” (p. 85), and the Discussion quietly introduces a fourth sample size — “n = 50, 100, 500, and 900” (p. 89) — that appears nowhere in the methods or in Table 18. The arithmetic resolves immediately: 108 = 4 × 3 × 3 × 3, so the only route to 108 is a fourth sample-size level, which is exactly the stray n = 900.

Show the arithmetic 
cat("Table 18 grid:  3 × 3 × 3 × 3 =", 3^4, "conditions\n")
#> Table 18 grid:  3 × 3 × 3 × 3 = 81 conditions
Show the arithmetic 
cat("Stated in text:                 108 conditions\n")
#> Stated in text:                 108 conditions
Show the arithmetic 
cat("108 = 4 × 3 × 3 × 3 =", 4 * 3^3, " → needs a 4th sample size (the stray n = 900)\n\n")
#> 108 = 4 × 3 × 3 × 3 = 108  → needs a 4th sample size (the stray n = 900)
Show the arithmetic 
# A "count" framing of the false-flag problem at the TWO-SIDED ±0.5-logit cut: how many
# NORMAL raters (true severity ~ N(0, 0.5)) are flagged by chance as too severe OR too
# lenient (|estimated λ| > 0.5, either direction)? This two-sided rate (~0.318) is ~2×
# Chapter 5's ONE-sided +0.5 false-positive rate (~0.159); the count below is R × 0.318.
fp <- data.frame(R = c(5, 10, 50))                      # raters in the dataset
fp$N    <- fp$R * 10                                    # examinees (raters = N/10)
fp$se   <- 1 / sqrt(fp$N * 5 * 0.77)                    # SE of a location (0.77 ≈ avg score variance W, cf. Ch.5/Ch.2)
fp$rate <- 2 * (1 - pnorm(0.5 / sqrt(0.5^2 + fp$se^2))) # P(|estimated λ| > 0.5)
fp$falsely_flagged <- round(fp$R * fp$rate, 1)
fp[, c("R", "rate", "falsely_flagged")]
#>    R      rate falsely_flagged
#> 1  5 0.3222868             1.6
#> 2 10 0.3198115             3.2
#> 3 50 0.3178128            15.9

At R = 50 raters, the table’s analytic expected count is roughly 16 normal raters tripping the ±0.5-logit flag by chance alone — where “flagged” means estimated too severe or too lenient, i.e. |λ| > 0.5 in either direction (a two-sided count). This is the legible “how-many-raters” companion to Chapter 5’s false-positive rate — but the two are not the same number wearing different clothes. Chapter 5’s ≈ 16% is a one-sided rate (the +0.5 tail of N(0, 0.5)), which on its own is ≈ 8 of 50 raters; counting flags in either direction roughly doubles the per-rater rate to ≈ 0.32, giving the ≈ 16 here. The matching “16”s are therefore a coincidence — Chapter 5’s 16 percent and this chapter’s 16 raters are different quantities that happen to land on the same digits.

Why it matters. Study 3 is entirely a simulation, so the condition grid is the empirical basis. Any inconsistency among what Table 18 lists, what the text claims, and what the ANOVAs actually processed means the results cannot be fully reproduced from the manuscript alone — readers cannot verify which cells fed the analyses, and the appendix tables become hard to interpret without knowing whether n = 900 was run.

Fix. Correct the count (to 81 or 108, whichever was actually run); add n = 900 to Table 18 if those cells were generated, or remove the Discussion reference if they were not; and release the condition log or simulation script so the grid is machine-readable and independently verifiable.

7.2 Two names for the Study 3 generating model

What happens. The Study 3 design states that ratings were generated from “a modified version of the MFRM that allows us to generate certain types of rater effects” (p. 84), written out as Equation (8) — a many-facet model with rater severity \(\lambda_j\), rater discrimination \(\alpha_j\), item difficulty \(\delta_i\), and thresholds \(\tau_{ik}\). Yet a few lines down on the same page, the local-independence factor is described as starting from ratings simulated under a different model: “After simulating baseline ratings using the GPCM, we randomly selected rater pairs…” (p. 84). The two phrases name two different data-generating models. The GPCM (Muraki, 1992) is an item-level polytomous model — item discrimination \(a_i\), no rater facets — whereas the adapted MFRM of Equation (8) is a many-facet model built around rater severity and rater discrimination. In the measurement literature these are not interchangeable labels for one model.

Why it matters. Study 3 is a simulation, so the generating model is the data. As written, a reader cannot tell whether the local-independence cells were built on the Equation (8) many-facet model the rest of the chapter implies, or on a genuinely different item-level GPCM — with the rater-pair dependence then injected on top of whichever baseline was used. That ambiguity blocks exact reproduction of the very condition this study is named for. It is almost certainly a wording slip rather than two real pipelines — the dissertation uses the GPCM legitimately elsewhere as the HRM’s first stage — but only a one-word correction can settle it for the reader.

Fix. State the generating model once, consistently, and use that name throughout Study 3. If (as is almost certain) Equation (8) is the intended baseline, replace “GPCM” on p. 84 with the adapted-MFRM / Equation (8) reference; if a separate GPCM baseline really was used for the local-independence factor, give its equation and say explicitly why it differs from Equation (8). Either way, one named model end-to-end removes the ambiguity at no cost to the findings.

7.3 Strengthening the ANOVA analysis

What happens. Each indicator is analyzed with a separate univariate ANOVA, and practical significance is declared when partial η² ≥ .01, on data with 100 replications per condition and multiple raters nested within each replication.

Why it matters. Three problems compound. First, raters within a replication share a common data-generating context and are not independent; the ANOVA treats them as if they were, understating standard errors. Second, the indicators are correlated outcomes analyzed in parallel with no multivariate test or multiplicity adjustment, which inflates the family-wise error rate invisibly. Third — and most consequential — with 100 replications × many raters, p-values are almost always significant, so the entire interpretation rests on the η² ≥ .01 bar, which (as Chapter 5 showed) corresponds to near-chance detection. The ANOVAs may be certifying “practically significant” factors whose practical contribution is negligible.

Fix. Analyze replicate-level summaries (collapsing within each replication first), or use a mixed-effects model with replication as a random factor — standard in simulation methodology and familiar to reviewers. Pre-specify one or two primary outcomes (or apply a multiplicity correction across the battery of ANOVAs), and report recovery/detection metrics alongside — or instead of — the omnibus F and η², so effect sizes can be read on a scale with direct practical meaning.

7.4 Coding reliability of the systematic review

What happens. The systematic review that opens the dissertation was coded by a single person: “I coded all articles based on the following criteria…” (p. 13), with disagreements “resolved through discussion with my Dissertation Chair until a consensus was reached” (pp. 13–14). No inter-coder reliability statistic — no Cohen’s κ, no percent agreement — is reported.

Why it matters. The systematic review’s contribution is its frequency tables characterizing the rater-effects literature, so their credibility rests on the coding. Let me be explicit that this is strictly additive, not fatal: the review is a genuine, keepable contribution — real work and real scholarship. But journal reviewers in measurement and applied linguistics routinely expect either a second independent coder with a reported reliability statistic on a sample of articles, or a documented coding audit; the current description gives no basis for assessing how consistently the criteria were applied.

Fix. Double-code a random ~20% sample of the articles independently, compute κ and percent agreement for each major category, and report them in the methods. If that is not feasible before submission, a transparent coding audit — a supplementary codebook documenting decision rules and borderline cases — will satisfy most reviewers.

TipFix-it
  1. Condition count. Reconcile the stated 108 with Table 18’s 81-cell grid; add n = 900 to Table 18 if those cells were run (or remove the reference); release the condition log.
  2. Study 3 generating model. State one model end-to-end: p. 84 names both the Equation (8) adapted MFRM and the GPCM as the baseline that produced the ratings. If Equation (8) is intended (almost certainly), drop the stray “GPCM”; otherwise give the GPCM baseline its own equation and rationale.
  3. ANOVA hygiene. Analyze replicate-level summaries (or mixed-effects models) to respect within-replication clustering; pre-specify primary outcomes or correct for multiplicity; report recovery/detection rates alongside F and η².
  4. Coding reliability. Double-code ~20% of the systematic-review articles and report Cohen’s κ / percent agreement, or supply a supplementary codebook documenting the decision rules.

And one reporting standard ties the simulation work together: attach a Monte-Carlo standard error to every number both studies average over the 100 replications — Study 2’s and Study 3’s condition means, detection/recovery rates, and AUC alike (the across-replication SD over \(\sqrt{100}\) for a mean, the binomial MC-SE for a rate), reported beside the quantity in the tables — so a reader can separate a real between-condition difference from simulation noise rather than reading 100-run averages as exact. It is the replication-level companion to the posterior MCSE Chapter 4 asks for at the estimation level, and generalizes the MC-SE Chapter 5 attaches to its detection metrics.

All four are fixable in an afternoon — none changes the dissertation’s central findings; each makes the manuscript meaningfully more defensible to a skeptical reviewer.
